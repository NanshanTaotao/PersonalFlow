# PersonalFlow 产品需求文档（PRD）

最后更新：2026-06-18

## 1. 产品定位

PersonalFlow 是一个本地优先的 AI 角色场景定制与演练产品。

用户可以用自然语言描述自己想要体验、练习或模拟的互动场景，PersonalFlow 会帮助用户生成一段可运行的个人化互动流程，并在对话中驱动多个 AI 角色按设定行动。会话结束后，产品可以基于过程生成结构化复盘，帮助用户理解表现、发现问题并继续改进。

PersonalFlow 的核心不是“和一个 AI 随便聊天”，而是：

```text
把用户想象中的角色、场景、流程和目标，变成一段可运行、可复盘、可再次调整的个人化互动体验。
```

产品初期以本地单用户产品为主，用户使用自己的 LLM API Key。MVP 不定位为托管式 SaaS，不做团队协作、商业化市场和复杂账号体系。

## 2. 产品愿景

PersonalFlow 希望成为用户的“私人互动场景定制器”。

用户不需要理解提示词工程、流程编排或技术配置，只需要描述：

```text
我想进入什么场景；
谁会参与；
他们分别是什么角色；
我想练习、体验或观察什么；
对话结束后我希望得到什么反馈。
```

产品负责把这些意图转成可运行的互动场景。

长期来看，PersonalFlow 可以覆盖：

- 求职面试、论文答辩、绩效沟通、销售谈判等练习场景；
- 辩论、模拟法庭、危机公关、客户沟通等专业演练；
- 剧本杀、侦探解谜、谁是卧底等角色扮演娱乐体验；
- 陪练、教练、导师、评审等个人成长场景；
- 任何需要多角色互动、过程控制和结果复盘的个人化模拟。

## 3. 产品目标

MVP 阶段的产品目标：

- 用户可以在本地配置自己的 LLM API Key 并开始使用。
- 用户可以用自然语言创建一个个人化互动场景。
- 产品能在信息不足时提出少量关键问题。
- 产品能展示用户可理解的场景预览，而不是强迫用户看配置文件。
- 用户确认后，可以进入真实互动会话。
- AI 角色能按照场景设定参与对话，而不是自由跑偏。
- 会话过程能展示当前场景进度、当前角色和用户该做什么。
- 会话结束后，用户可以获得一份结构化复盘。
- 产品内置少量高质量模板，用户可以一键开始。
- 高级用户可以查看、复制、导入和导出场景配置，但这不是普通用户必经路径。

## 4. 非目标

MVP 不做：

- 托管式多租户 SaaS。
- 账号系统、团队空间、权限体系和计费。
- 移动端优先体验。
- 音视频实时演练。
- 多人真人在线协作。
- 场景市场、插件市场或商业模板交易。
- 完整角色知识库和深度材料检索系统。
- 复杂可视化流程图编辑器。
- 任意外部工具调用平台。
- 要求普通用户手写 JSON、YAML 或代码。

## 5. 目标用户

### 5.1 普通用户

普通用户希望快速创建并体验一个个人化互动场景。

典型需求：

- 我要练习一次系统设计面试。
- 我要模拟一次论文答辩，评委可以尖锐一点。
- 我要练习和老板谈晋升。
- 我要体验一个侦探解谜场景。
- 我要让 AI 扮演客户，帮我练销售话术。

他们期望：

- 不写配置。
- 不学习复杂概念。
- 用自然语言说明需求。
- 快速开始对话。
- 得到有帮助的复盘。

### 5.2 高级场景作者

高级场景作者希望更精细地控制场景。

他们期望：

- 查看完整场景配置。
- 复制已有模板继续修改。
- 直接编辑角色、流程、复盘维度和材料。
- 导入或导出场景文件。
- 让 AI 解释当前场景设置。
- 让 AI 根据自然语言修改已有场景。

### 5.3 模板贡献者

模板贡献者希望创作可复用的场景模板。

他们期望：

- 模板结构清晰。
- 用户参数简单。
- 模板运行行为可预期。
- 模板可以被复制、分享和迭代。
- 产品能检查模板是否可运行。

## 6. 核心概念

### 6.1 场景

场景是用户想要体验或练习的一段互动过程。

它包含：

- 场景目标。
- 参与角色。
- 用户扮演谁。
- AI 扮演谁。
- 大致流程。
- 需要使用的材料。
- 对话结束条件。
- 是否需要复盘。

产品面向普通用户时，优先展示“场景预览”，而不是底层配置。

### 6.2 角色

角色是场景中的行动者。

角色可以是：

- 用户本人。
- AI 面试官。
- AI 评委。
- AI 客户。
- AI 对手。
- AI 主持人。
- AI 玩家。
- AI 观察者或复盘者。

每个角色可以有自己的目标、性格、沟通风格、知道的信息和行为边界。

### 6.3 流程

流程描述互动如何推进。

例如：

- 面试从自我介绍开始，然后进入多轮提问，最后给出总结。
- 答辩先陈述论文，再由评委提问，最后进行综合评价。
- 谁是卧底先分配词语，再轮流描述，然后投票淘汰。
- 谈判先确认诉求，再交换条件，最后尝试达成协议。

普通用户不需要手写流程，产品通过模板和 AI 帮助生成。

### 6.4 模板

模板是一类可复用场景的起点。

MVP 内置模板建议包括：

- 求职面试。
- 论文答辩。
- 销售沟通。
- 绩效沟通。
- 自由角色扮演。

每个模板提供合理默认值，用户可以直接运行，也可以先定制。

### 6.5 场景助手

场景助手是帮助用户创建和修改场景的 AI 助手。

它负责：

- 理解用户想要什么。
- 判断信息是否足够。
- 只追问必要问题。
- 生成场景草稿。
- 解释场景预览。
- 根据用户反馈修改场景。
- 帮用户修复无法运行的问题。

场景助手不应该一次性抛出复杂配置，也不应该无休止追问。

### 6.6 演练会话

演练会话是用户进入场景后的正式互动过程。

在会话中：

- 产品展示当前场景进度。
- AI 角色按设定发言。
- 用户按自己的角色参与。
- 产品在必要时推动流程。
- 用户可以暂停、继续或结束。

### 6.7 复盘

复盘是会话结束后的反馈报告。

它可以包括：

- 本次互动总结。
- 用户表现亮点。
- 需要改进的问题。
- 关键对话片段。
- 分维度评价。
- 下一次练习建议。

复盘应尽量引用会话中的具体证据，而不是泛泛而谈。

## 7. 产品原则

- 自然语言优先，配置其次。
- 普通用户不需要理解底层结构也能使用。
- 高级用户可以展开细节，但默认不打扰普通用户。
- 产品应该给出合理默认值，而不是要求用户填完所有字段。
- AI 角色可以灵活表达，但不能随意改变场景规则。
- 用户始终知道当前在什么场景、轮到谁、接下来要做什么。
- 复盘必须尽量基于实际对话，而不是只给模板化鼓励。
- 会话历史不应被静默覆盖；后续需要支持从历史节点重新尝试、撤回或创建分支。
- 本地优先，用户清楚自己的数据保存在哪里。
- 模板应该可复制、可修改、可分享。

## 8. 核心用户旅程

### 8.1 首次使用

1. 用户打开 PersonalFlow。
2. 产品引导用户配置 LLM API Key。
3. 用户测试连接。
4. 产品确认可用后进入首页。
5. 用户可以选择模板或直接描述想要的场景。

### 8.2 使用模板快速开始

1. 用户进入首页。
2. 用户选择一个模板，例如“求职面试”。
3. 产品展示模板预览。
4. 用户可以直接开始，也可以先修改难度、角色风格、主题和复盘维度。
5. 用户确认后进入演练会话。
6. 会话结束后查看复盘。

### 8.3 自然语言创建场景

1. 用户输入：“我想模拟一次 B 端 SaaS 销售谈判，对方比较强势，我想练习如何守住价格底线。”
2. 场景助手识别用户目标、参与角色和场景类型。
3. 如果信息不足，场景助手最多追问少量关键问题。
4. 用户回答，或要求产品使用默认值。
5. 产品生成场景草稿。
6. 产品展示可读预览。
7. 用户确认后进入演练。

### 8.4 修改场景

用户可以说：

- “让面试官更关注系统设计。”
- “评委可以更尖锐一点，但不要攻击人格。”
- “把客户改成预算紧张但有明确需求。”
- “增加最后的量化评分。”
- “这次不要太长，控制在 10 分钟内。”

产品应保留用户已有设定，只修改用户要求的部分，并展示更新后的预览。

### 8.5 运行演练

1. 用户进入会话页面。
2. 产品展示当前场景、用户角色、当前阶段和当前发言者。
3. AI 角色按场景发起或回应。
4. 用户输入内容或执行场景要求的动作。
5. 产品按流程推进。
6. 用户可以暂停、继续、结束。

### 8.6 查看复盘

1. 会话结束。
2. 用户点击“生成复盘”。
3. 产品生成报告。
4. 用户查看总结、维度评价、关键片段和建议。
5. 用户可以基于复盘重新练习，或修改场景后再试一次。

## 9. 功能范围

### 9.1 本地模型配置

MVP 需要支持：

- 新增 LLM 配置。
- 填写 API Key、Base URL、模型名称。
- 测试连接是否可用。
- 设置默认模型。
- 编辑和删除配置。
- API Key 不以明文形式重复展示。

### 9.2 首页

首页需要提供：

- 创建新场景。
- 从模板开始。
- 继续最近草稿。
- 查看最近会话。
- 导入场景文件。
- 进入设置。

### 9.3 模板库

MVP 模板至少包含：

- 求职面试。
- 论文答辩。
- 自由角色扮演。

模板卡片展示：

- 名称。
- 简短描述。
- 适合练习什么。
- 参与角色。
- 预计时长。
- 是否支持复盘。

### 9.4 场景创建

用户可以通过自然语言创建场景。

场景助手需要识别：

- 场景类型。
- 用户目标。
- 用户扮演的角色。
- AI 需要扮演的角色。
- 互动风格。
- 大致时长。
- 是否需要材料。
- 是否需要复盘。

追问规则：

- 只追问影响场景运行的关键信息。
- 初始创建阶段最多追问 5 个问题。
- 超过限制后使用合理默认值。
- 默认值应在预览中明确标注。

### 9.5 场景预览

预览需要展示：

- 场景名称。
- 场景目标。
- 用户角色。
- AI 角色。
- 互动流程。
- 预计时长。
- 难度或压力程度。
- 复盘方式。
- 使用的材料。
- 当前是否可以开始。

普通用户默认看到预览，不看到底层配置。

### 9.6 场景检查

产品需要在场景生成和修改后检查是否可以运行。

检查结果分为：

- 可以开始。
- 可以开始但有提醒。
- 需要修复。

检查信息必须面向用户表达。

示例：

```text
需要修复：这个场景还没有设置用户扮演的角色，因此无法开始。
建议：将用户设为“候选人”，AI 设为“面试官”。
```

### 9.7 场景确认

用户点击开始前，需要看到确认页或确认状态。

确认内容包括：

- 场景摘要。
- 角色列表。
- 互动流程。
- 复盘设置。
- 模型配置是否可用。
- 需要用户注意的事项。

确认后，当前场景成为本次会话使用的固定版本。后续编辑草稿不会影响已经开始的会话。

### 9.8 演练会话

会话页面需要支持：

- 展示对话记录。
- 展示当前场景和阶段。
- 展示当前发言者。
- 展示用户当前可做的动作。
- 用户输入消息。
- AI 角色回复。
- 暂停和继续。
- 手动结束。
- 会话结束后进入复盘。

MVP 不要求同时展示复杂流程图。

### 9.9 多角色互动

MVP 至少支持：

- 一个用户角色。
- 一个或多个 AI 角色。
- AI 角色按场景顺序参与。
- 用户只能看到当前会话中应该看到的信息。

后续可以扩展到：

- 多个真人用户。
- 更复杂的私密信息。
- 游戏化角色机制。

### 9.10 材料

MVP 可以支持轻量材料：

- 用户粘贴文本材料。
- 用户给材料命名。
- 场景助手可以参考材料创建场景。
- 演练角色可以在允许范围内引用材料。
- 复盘可以引用材料相关表现。

MVP 不要求复杂文件解析和知识库。

### 9.11 复盘

复盘报告需要包含：

- 总体总结。
- 分维度评价。
- 关键片段。
- 具体建议。
- 下一次练习建议。

可选包含：

- 评分。
- 风格分析。
- 表达清晰度评价。
- 目标达成度。

复盘应尽量引用会话原文。

### 9.12 场景管理

用户可以：

- 查看草稿。
- 继续编辑草稿。
- 复制场景。
- 删除场景。
- 导入场景文件。
- 导出场景文件。
- 查看历史会话。
- 查看历史复盘。

## 10. MVP 范围

MVP 必须包含：

- 本地 LLM API Key 配置与测试。
- 首页和模板入口。
- 至少 3 个内置模板。
- 场景预览。
- 场景检查。
- 场景确认并开始。
- 单用户演练会话。
- 一个用户角色 + 一个或多个 AI 角色。
- 暂停、继续、结束。
- 手动触发复盘。
- 场景草稿、本次会话和复盘本地保存。
- 场景导入与导出。

MVP 可以暂缓：

- 自然语言创建场景。
- 场景助手关键追问。
- 音视频。
- 移动端适配。
- 模板市场。
- 复杂材料解析。
- 可视化流程图编辑。
- 深度角色知识库。
- 自动生成多版本对比报告。
- 完整娱乐游戏模板库。
- 场景分支、撤回、重试和 fork 的完整可视化体验。

MVP 明确不做：

- 多真人在线协作。

多真人参与不是简单增加在线用户。真人输入不可预测，产品需要处理多人意图、冲突、超时、可见性、流程恢复和责任边界。它会显著改变场景运行方式，应放到更远版本作为架构级升级，而不是作为 MVP 后的小功能。

## 11. 关键页面

### 11.1 设置页

用于配置 LLM。

需要展示：

- 当前模型配置。
- 连接测试状态。
- 默认模型。
- API Key 管理说明。

### 11.2 首页

用于开始使用。

需要展示：

- 新建场景入口。
- 模板入口。
- 最近草稿。
- 最近会话。
- 设置入口。

### 11.3 场景创建页

用于和场景助手对话。

需要展示：

- 用户与场景助手的对话。
- 当前草稿状态。
- 场景预览。
- 检查结果。
- 修改建议。
- 开始按钮。

### 11.4 场景确认页

用于让用户在开始前确认。

需要展示：

- 场景摘要。
- 角色。
- 流程。
- 材料。
- 复盘方式。
- 提醒和风险说明。

### 11.5 演练页

用于正式互动。

需要展示：

- 对话区。
- 当前场景状态。
- 当前角色。
- 用户输入区。
- 暂停、继续、结束按钮。
- 复盘入口。

### 11.6 复盘页

用于查看报告。

需要展示：

- 总结。
- 维度评价。
- 关键片段。
- 建议。
- 重新练习入口。
- 基于复盘修改场景入口。

## 12. 成功标准

MVP 成功标准：

- 新用户可以在 5 分钟内配置模型并运行一个模板场景。
- 用户可以通过内置模板创建并确认一个可运行场景。
- 普通用户完成一次演练不需要查看或编辑配置文件。
- 场景预览能让用户理解即将发生什么。
- 演练过程中用户知道当前轮到谁、自己要做什么。
- AI 角色大体遵守场景设定和互动流程。
- 会话结束后用户能获得一份有具体建议的复盘。
- 用户愿意基于复盘再次练习或修改场景。

## 13. 体验质量要求

- 产品文案避免技术术语，优先使用“场景”“角色”“流程”“复盘”等用户可理解词汇。
- 错误提示应说明问题和建议，而不是只展示失败码。
- 场景创建过程不应让用户感觉像在填表。
- 默认值应该让用户能快速开始，但不能偷偷改变用户明确表达的意图。
- 复盘应具体、克制、有证据，避免空泛夸奖。
- 高级配置入口可以存在，但默认折叠。
- 本地数据和 API Key 使用方式需要清楚说明。

## 14. 风险与应对

### 14.1 场景创建过于复杂

风险：用户需要回答太多问题，创建成本过高。

应对：

- 限制关键追问数量。
- 使用模板和默认值。
- 先生成可运行草稿，再允许逐步修改。

### 14.2 AI 角色跑偏

风险：AI 角色没有按场景设定行动。

应对：

- 产品在会话中展示当前阶段和角色任务。
- 场景设定应尽量明确角色目标和边界。
- 对明显跑偏的回复提供重试或继续机制。

### 14.3 复盘不可信

风险：复盘泛泛而谈，用户不相信。

应对：

- 复盘尽量引用具体对话片段。
- 明确区分事实、推断和建议。
- 对不确定判断保持克制。

### 14.4 本地配置门槛

风险：用户不会配置 API Key。

应对：

- 提供清晰设置指引。
- 提供连接测试。
- 对常见错误给出可读说明。

### 14.5 产品边界过宽

风险：同时做练习、游戏、创作、复盘，导致 MVP 发散。

应对：

- MVP 先聚焦“创建场景 -> 运行演练 -> 生成复盘”闭环。
- 模板数量少但质量高。
- 娱乐类复杂玩法放到后续版本。

## 15. 后续版本方向

Post-MVP 可以考虑：

- 更多模板：谈判、模拟法庭、危机沟通、剧本杀、谁是卧底。
- 场景分享和社区模板库。
- 可视化流程预览。
- 更强的材料处理能力。
- 更细的复盘维度和历史对比。
- 角色长期记忆和个人化陪练。
- 场景分支、撤回、重试和 fork 的可视化体验。
- 更丰富的导入导出格式。

更远期方向：

- 多真人参与和多人协作场景。

多真人参与需要重新设计会话控制、用户输入解释、冲突处理、权限可见性和恢复机制，不能按单用户演练的自然延伸处理。

## 16. 一句话总结

PersonalFlow 是一个把个人意图转成可运行互动场景的 AI 产品。

它让用户可以像“私人订制”一样描述自己想要的角色和情境，然后进入一段由 AI 角色参与、流程可控、结束后可复盘的个人化体验。
